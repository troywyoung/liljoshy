# Brand Voice Guidelines

Dust's brand voice is direct, technical, practical, and confident — like a Unix utility.

## Voice Attributes

**Direct**: Say what it does, not what it might enable. No marketing speak.

**Technical**: Developer-to-developer communication. Use precise terminology without dumbing down.

**Practical**: Focus on workflow, not hype. Show real commands, real outcomes.

**Confident**: Dust knows what it is. No hedging, no over-promising.

## Good vs. Bad Examples

### Taglines and Descriptions

✅ "Flow state for AI coding agents."
❌ "Unlock your team's potential with cutting-edge AI collaboration."

✅ "Dust structures your project so agents can understand what needs to be done and execute systematically."
❌ "Dust seamlessly empowers AI agents to revolutionize your development workflow."

### Command Descriptions

✅ "Initialize a new Dust repository"
❌ "Set up your project to leverage the power of Dust"

✅ "Show tasks ready to work on (not blocked)"
❌ "Discover actionable opportunities for your next sprint"

### Error Messages

✅ "Error: .dust directory not found\nRun 'dust init' to initialize a Dust repository"
❌ "Oops! It looks like you haven't set up Dust yet. Try running dust init!"

✅ "Error: No objective provided\nUsage: dust focus \"your objective here\""
❌ "We couldn't find an objective. Please provide one to continue."

### Status Messages

✅ "Validating directory structure..."
❌ "Hang tight while we check everything..."

✅ "All validations passed!"
❌ "Great news! Everything looks perfect!"

### Workflow Instructions

✅ "Pick a task → implement it → delete the task file → commit atomically."
❌ "Choose a task that inspires you, bring your changes to life, clean up, and save your progress!"

### Technical Concepts

✅ "Each commit tells a complete story, bundling implementation changes with their corresponding documentation updates."
❌ "Our atomic commit approach ensures your codebase stays healthy and maintainable over time."

✅ "AI agents need structure to work effectively."
❌ "Empower your AI agents to reach their full potential with structured workflows."

## Context-Specific Guidance

### CLI Output
- Use minimal, tasteful emojis as section markers (✨, 🤖, 💡, 📋, 🎯, 🔍)
- Keep output scannable with clear visual hierarchy
- Error messages must be actionable (see `actionable-errors.md` principle)
- Status messages should be brief, present tense, technical

### Documentation
- Lead with what it does, not why you need it
- Use concrete examples over abstract descriptions
- Show real commands and workflows
- Avoid hedge words: "might", "could potentially", "helps to"

### Error Messages
- State the problem clearly
- Provide the exact command to fix it
- Use technical terminology without explanation

### Marketing Copy
- Forbidden words: "revolutionize", "unlock", "empower", "cutting-edge", "seamless", "robust", "powerful"
- Focus on concrete benefits: "enables X to do Y" not "empowers teams to achieve more"
- Lead with the tagline: "Flow state for AI coding agents"

## Positioning Reference

**Brand Style**: Minimal, Tool-Focused (Unix utility)
**Audience**: Developers using AI coding tools
**Tone**: Technical, confident, no-nonsense

This positioning informs all voice decisions. When in doubt, ask: "Would this fit in a Unix man page?"
